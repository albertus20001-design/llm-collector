# 模型选择框架

这一页先讲共性，不先按厂商拆。

## 一、先看任务类型

优先判断任务属于哪一类：

- 聊天 / 问答
- 代码生成 / 调试
- 结构化输出 / 工作流
- 长文档理解
- 多模态理解
- 智能体 / 工具链编排

不同任务最怕的不是“模型不够强”，而是模型能力和任务约束不匹配。

## 二、再看真正的约束条件

常见约束：

- 延迟
- 成本
- 上下文长度
- 工具调用稳定性
- 权限 / 合规 / 区域可用性
- 是否必须走某个供应商 / 平台

## 三、不要只看基准测试

更实用的判断维度：

- API 是否稳定
- 是否支持你要的结构化输出方式
- 工具调用是否可控
- 长上下文是否真的稳定
- 限流是否影响落地
- 单价和实际可用成本是否一致

## 四、推荐回答结构

如果用户问“哪个模型更适合”，建议这样回答：

1. 先问任务类型
2. 再问是否有成本 / 延迟 / 上下文约束
3. 再给 2–3 个可选项
4. 最后说明各自取舍

## 五、最常见误区

- 把“最强模型”误认为“最适合模型”
- 只看单轮效果，不看工作流稳定性
- 忽略供应商 / 聚合平台差异
- 忽略结构化输出和工具兼容性
