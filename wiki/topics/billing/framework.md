# 计费框架

这一页先讲共性，不先按厂商拆。

## 一、先分清“计什么”

常见计费对象：

- **输入 token**
- **输出 token**
- **缓存写入 / 缓存读取**
- **工具调用**
- **多模态输入**
- **长上下文处理**
- **并发 / 速率 / 层级资源**

## 二、再分清“在哪里计”

通常有三层：

1. **原厂 API 价格**
2. **聚合平台价格**
3. **内部实际落地成本**（重试、缓存命中率、失败率、工具链成本）

同一个模型，在这三层看到的成本可能并不一样。

## 三、比较时不要只看单价

更稳的比较维度：

- 输入 / 输出价格是否对称
- cache 是否能显著摊薄成本
- 是否存在长上下文加价
- 工具调用是否额外收费
- 速率限制是否会逼迫多账号 / 多 provider / 更贵方案
- 聚合平台是否因为 provider 路由而出现价差

## 四、回答计费问题的标准结构

建议按这个顺序回答：

1. **先给共性定义**：这类费用一般怎么构成
2. **再给厂商差异**：谁支持、谁没有、谁写得更清楚
3. **最后给决策提示**：什么场景下该优先看单价，什么场景下该优先看总成本

## 五、最常见误区

- 只比较输入 token 单价，不比较输出 token
- 把原厂价格和聚合平台价格混在一起
- 忽略 cache 命中率对总成本的影响
- 忽略速率限制导致的“理论便宜、实际不够用”
- 把工具能力当成“免费附赠”，没有单独评估其成本
